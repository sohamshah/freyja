01Introduction

The dominant paradigm in AI-assisted software development treats the agent as a stateless tool — each session starts from scratch, context is managed implicitly by the provider, and the agent’s capabilities are fixed at deployment time. This works tolerably for short coding tasks but breaks down for the kind of sustained, multi-modal knowledge work that a desktop agent should support: researching across documents and APIs, drafting communications, automating repetitive GUI workflows, and gradually learning a user’s preferences and project conventions.

Three problems motivate Freyja’s architecture:

Context is scarce
Even 200K-token windows fill quickly when tool schemas, file contents, and conversation history compete for space. The MemoryArena benchmark demonstrates that models near-perfect on passive recall “plummet to 40–60%” on interdependent multi-session agentic problems. Long context is not memory.
Skills are fragile
Hand-written instructions degrade as codebases evolve, and without a feedback loop the agent has no mechanism to distinguish helpful instructions from harmful ones.
Tool explosion is real
Connecting three typical MCP servers (GitHub, Slack, Sentry) consumes approximately 94,000 tokens of schema overhead — 47% of a 200K window — before any conversation begins, as measured by MindStudio.

We address these problems with a layered architecture whose components are described in the sections that follow. Where possible, we adopt existing standards and patterns rather than inventing new ones, and we note explicitly where our design diverges from prior work and why.

02Related Work

Our design draws on several families of prior work, which we summarize here to clarify what we adopt, what we extend, and what we depart from.

Agent frameworks

Claude Code provides the closest reference architecture: a while(tool_call) loop with no DAGs or classifiers, eight core tools, a skill system based on Markdown files with YAML frontmatter, and a compaction mechanism that fires at approximately 83.5% of context capacity. We adopt its permission model, skill format, and compaction survival rules essentially unchanged, extending them with confidence tracking and evolutionary optimization. OpenHands contributes the Condenser abstraction — a pluggable interface for context compression with multiple implementations (recent-events, LLM-summarizing, amortized-forgetting) — which informed our layered eviction design. Cursor takes a fundamentally different approach, treating each conversation as short-lived and relying on codebase indexing to re-fetch context on demand rather than compressing a growing transcript; we consider this a valid alternative for IDE-integrated tools but unsuitable for a long-running autonomous agent. Devin operates as a compound AI system with specialized models for planning, coding, review, and browsing, which validates our multi-model routing approach but requires full cloud infrastructure that we avoid.

Skills and instruction packages

The agentskills.io open standard, adopted by 30+ agent tools including Claude Code, Cursor, Gemini CLI, and GitHub Copilot, defines a SKILL.md format with YAML frontmatter and a Markdown body, optional supporting files, and a progressive disclosure pattern (description at discovery, full body on activation). Hermes Agent extends this standard with a self-evolution pipeline that reads execution traces, diagnoses failures, generates candidate mutations via GEPA, and gates promotion through five mandatory constraint checks. We adopt both the standard and the evolution pipeline, adapting the latter to our confidence lifecycle.

Episodic memory

MemRL demonstrates that a frozen LLM augmented with an evolving memory bank of Intent-Experience-Utility (IEU) triplets, where utility scores are updated via exponential moving average after each episode, achieves a 56% gain on multi-step tasks (ALFWorld) and a forgetting rate of 0.041 versus 0.051 for the baseline. The two-phase retrieval mechanism — similarity-based recall followed by value-aware ranking that downweights semantically similar but historically unsuccessful memories — is the key differentiator from naive RAG, and we adopt it for our episode retrieval system.

Prompt optimization

GEPA (ICLR 2026 Oral) evolves textual parameters using LLM-based diagnostic reflection and Pareto-efficient search, outperforming GRPO by 6% on average (up to 20%) while requiring 35x fewer rollouts, at a cost of approximately $2–10 per optimization run. DSPy provides the compilation framework; its core lesson — that bootstrapping from successful execution traces is the most reliable way to improve prompts — directly informs our skill mutation operators.

Context management

Factory.ai demonstrates that structured anchored summarization (3.70/5 quality) outperforms both Anthropic’s (3.44/5) and OpenAI’s (3.35/5) approaches across 36K production messages. JetBrains (NeurIPS DL4Code 2025) shows, perhaps counterintuitively, that observation masking — replacing old tool outputs with [details omitted] rather than summarizing — achieves +2.6% solve rate on SWE-bench Verified with 52% cost reduction. The BATS budget-awareness signal reduces speculative searches by 40.4% at near-zero cost. We combine all three techniques.

Codebase understanding

Aider contributes the most rigorous approach to repository mapping: a tree-sitter-parsed directed graph ranked via personalized PageRank, serialized within a configurable token budget. Aider’s edit format research found that unified diffs reduce LLM laziness by 3x versus search/replace blocks, and that function-call-based edit formats consistently underperform plain-text formats. Their architect mode, which separates planning from editing, produced state-of-the-art results (85% with o1-preview + DeepSeek). Augment Code builds a custom semantic graph across 400,000+ files, exposed as an MCP server usable by any agent.

Agent specialization

ForgeCode implements a three-agent architecture — FORGE (implementation), MUSE (planning, writes only to plans/), and SAGE (internal research, read-only) — where the analytical agent cannot trigger code changes. Roo Code extends Cline with custom modes that scope tool permissions per persona via regex patterns, and a “Boomerang Tasks” hub-and-spoke sub-agent model architecturally similar to our episode model. Slate (Random Labs, YC) is the most explicit implementation of the thread-weaving pattern we adopt: an orchestrator dispatches parallel worker threads, each of which produces an episode containing only successful tool calls and conclusions.

Sandboxing and generation

Bolt.new parses structured XML tags from the LLM stream into file writes and shell commands that execute progressively as tokens arrive. v0 (Vercel) uses a composite model pipeline achieving 93% error-free generation versus ~62% baseline. E2B provides Firecracker microVM sandboxes with ~150ms boot time and hardware-level isolation. Codex CLI implements kernel-level sandboxing via macOS Seatbelt and Linux Landlock.

03Architecture Overview

Freyja’s architecture is organized around a unified memory layer that mediates between persistent stores (skills, episodes, preferences, project instructions) and the finite context window of the LLM on each turn. We treat the context window as scarce RAM with a layered eviction strategy, rather than as an append-only log that eventually overflows into compaction.

Freyja · Memory Layer Architecture

Following Google ADK’s three-layer context model, we distinguish between the session (the durable, append-only event log that serves as ground truth), the working context (the computed projection of the session into a single model invocation’s prompt), and artifacts (large payloads that live in external stores with lightweight handles in context). This separation enables multiple optimization strategies — compaction, caching, filtering, artifact externalization — without losing data.

The engine itself is decoupled via a submission/event queue abstraction, following Codex CLI’s architecture: operations flow through a sequential processing loop with bounded channels, enabling multiple simultaneous UIs (desktop app, CLI, IDE extension), session resumability, and testable cores without coupling the agent logic to any particular frontend.

04Skills

The Externalization survey (Zhou et al., April 2026) defines a skill as the externalization of procedural expertise — not a tool (which exposes an operation) and not a protocol (which governs invocation), but a higher-level artifact encoding how a class of tasks should be executed. This expertise has three coupled components: the operational procedure (step decomposition, ordering, stopping conditions), decision heuristics (what to try first at branch points, when to back off), and normative constraints (testing requirements, scope limits, domain rules). Once externalized, skills become carriers of governance as much as carriers of capability.

We instantiate this framework as SKILL.md files with YAML frontmatter and Markdown bodies, adopting the agentskills.io open standard (30+ agent tools) and extending it through the metadata.freyja namespace.

Retrieval at scale

Our current retrieval (keyword triggers + LLM auto-discovery) is adequate for small catalogs but will not scale. SkillFlow demonstrates that over 36K skills, keyword retrieval achieves MRR 0.266 while a four-stage progressive pipeline achieves MRR 0.634. More critically, AgentSkillOS proves that flat skill invocation fails catastrophically at 200K scale:

Configuration200 skills1K skills200K skills
DAG orchestration (Quality-First)100.0100.0100.0
DAG orchestration (Efficiency-First)58.576.189.0
Flat invocation (full pool)24.348.117.2
Vanilla (no skills)0.00.00.0
Flat invocation gets worse at scale
Even with oracle-perfect skill selection, flat invocation (loading skills without modeling dependencies or execution order) produces lower quality than DAG orchestration. The quality gap is structural. Skills declaring depends_on and composes_with relationships, enabling dependency-ordered execution with parallelism within layers, is essential for scaling.

Skill composition

The Externalization survey identifies five composition patterns — serial execution, parallel division of labor, conditional routing, recursive sub-skill invocation, and pipeline chaining — and notes that repeatedly validated compositions can themselves be packaged as new higher-level skills. CUA-Skill formalizes this as composition graphs G_c = (V_c, E_c) where each node is a skill and each directed edge represents a valid chaining from one skill to another. This is directly analogous to our episode composition model (§5), but at the skill level — and the two should compose: a skill’s execution graph determines what sub-agents to spawn, and those sub-agents produce episodes that feed downstream skills.

Meta-cognitive activation

Act Wisely (Yan et al., April 2026) demonstrates that standard tool-augmented agents invoke tools on 80–98% of queries, but a meta-cognitive approach — where the agent first assesses whether it genuinely needs external computation — reduces invocation to ~2% while improving accuracy (+26.4% on WeMath). The implication: the agent should assess whether it can handle the task from parametric knowledge before loading a skill or calling a tool. This converts the skill selection problem from “which skill matches?” to “do I need a skill at all?” — and only entering the selection pipeline when the answer is yes dramatically reduces both context cost and selection errors.

Skill security

The Externalization survey warns that skill files themselves can become prompt-injection surfaces, and large-scale studies of public skill ecosystems report substantial rates of vulnerabilities including data exfiltration, privilege escalation, and supply-chain risk. This motivates our allow/deny permission model (§10) and argues for sandboxing skill-provided scripts, treating third-party skills with the same caution as third-party npm packages.

Cross-user evolution

SkillClaw (Ma et al., April 2026) demonstrates that a dedicated evolver agent — separate from the main session — can aggregate trajectories from multiple users, perform causal chain analysis, and generate targeted skill mutations achieving +42.1% average improvement. Key adoptable patterns: a client proxy that transparently intercepts API calls for trajectory collection, evidence-backed versioning citing specific sessions and scores, and a nighttime validation gate running head-to-head comparisons before promotion.

05Episodes and Episodic Memory

When a sub-agent completes its work, it produces an episode — a structured summary of what was accomplished, what was learned, and what files were touched — rather than passing the full transcript back to the parent. This design, inspired by Slate’s “thread weaving” architecture, allows the parent session to benefit from sub-agent work without inheriting its context cost.

Specimen · Episode record (JSON)
{
  "id": "ep_19d812345",
  "goal": "Research the caching architecture",
  "agent_id": "explore",
  "model": "claude-haiku-4-5",
  "status": "success",
  "duration_ms": 12000,
  "tokens_in": 5200,
  "tokens_out": 1800,
  "tool_calls": 12,
  "summary": "Redis for sessions, LRU for API responses. Hit rate ~60%.",
  "key_findings": [
    "Redis pool at src/cache/redis.ts:45 — max 10 connections",
    "No cache warming on deploy — cold start ~30s"
  ],
  "files_examined": ["src/cache/redis.ts", "src/cache/lru.ts"],
  "utility_q_value": 0.5,
  "importance_score": 0.85,
  "tags": ["caching", "redis", "performance"]
}
MemRL reinforcement
The utility_q_value enables MemRL-style learning: after an episode is reused, its Q-value is updated via exponential moving average (Q_new = Q_old + α(r − Q_old)). Retrieval uses two phases — similarity recall, then value-aware ranking with score (1−λ) · similarity + λ · Q_value — which downranks semantically similar but historically unsuccessful episodes.

06Agent Orchestration

The main agent is what the user talks to: it maintains the conversation transcript, owns the session context, decides when to delegate to sub-agents, synthesizes episode results, manages tool approval, and persists the session. Sub-agents are spawned for bounded tasks, receiving a fresh context, a goal, relevant episodes, and relevant skills. They produce an episode on completion and cannot spawn further sub-agents (single-level nesting).

07Model Router

We define five named model slots — main, subagent, search, reasoning, and computer — each configurable by the user. This is not an arbitrary decomposition: the same four-slot structure (minus computer) emerged independently in Slate, Claude Code, and Aider. We add a fifth slot for computer control, which requires vision capabilities that the other slots do not.

08Tool Management and MCP

Freyja ships with ~30 built-in tools. Users extend this set via MCP servers. The challenge is that tool schemas are expensive (300–800 tokens each) and the RAG-MCP benchmark demonstrates a sharp accuracy cliff above 200 tools.

Before entering the tool selection pipeline at all, the agent should assess whether it genuinely needs a tool — the meta-cognitive gate described in §4. Act Wisely shows that this pre-filter alone eliminates 96–98% of queries from the selection problem, converting the scaling challenge from “select among 200+ tools for every query” to “select among 200+ tools for the small fraction of queries that genuinely need tools.”

09Context Management

The context refill problem — where compaction triggers repeatedly because the summary silently lost exact details that the model then re-fetches — is the primary production pathology in long-running agent sessions. We combine four techniques, applied in priority order so that cheaper interventions fire first:

PriorityTechniqueSourceMeasured impact
1Prevention — snippets not whole files; compact diffs not full rewritesMorphSessions 15–30 min → 1–3 hours
2Budget signal — [context: 47K/131K — MEDIUM]BATS−40.4% speculative searches
3Observation masking — stubs not summariesJetBrains+2.6% solve rate, −52% cost
4Anchored summarization — mandatory sectionsFactory.ai3.70/5 vs Anthropic 3.44
Key finding
JetBrains shows that observation masking — which makes no attempt to preserve information — outperforms LLM summarization, which actively introduces errors by hallucinating details. Stubs are honest about what they don’t contain; summaries pretend.

14Future Directions

Several aspects of the architecture remain underspecified or unexplored, which we outline here both as a record of known limitations and as an invitation for future work.

Codebase understanding. We lack a dedicated repo-mapping mechanism. Aider’s tree-sitter + PageRank approach and Augment Code’s semantic graph (available as MCP server) are candidates for integration.

Edit format selection. Aider’s research shows the optimal edit format varies by model, and function-call formats consistently underperform plain text. We should evaluate model-specific format selection and the architect-mode pattern.

OS-level sandboxing. Codex CLI’s kernel-level enforcement (Seatbelt, Landlock) provides stronger guarantees than our application-level approach.

Artifact externalization. Google ADK’s handle pattern — large payloads in stores with lightweight refs in context — is a cleaner solution to tool-result bloat than head/tail truncation.

Agent observability. We have no tracing infrastructure. AgentOps’s nine agent-specific span types and Braintrust’s production-trace-to-test-case pipeline should be adopted from Phase 1.

Streaming execution. Bolt.new’s streaming parser could reduce perceived latency for multi-file generation, but compatibility with our permission model requires investigation.

Orchestrator-never-executes. Slate and ForgeCode enforce that the orchestrator only reasons and dispatches. Whether Freyja’s main agent should adopt this constraint is an open design question.

Cross-platform support. macOS-only due to Rust native extension. Linux (AT-SPI) and Windows (UI Automation) need platform-specific backends behind a shared trait interface.

Privacy in computer control. Screenshot redaction (AXSecureTextField, password manager exclusion) is partial. Comprehensive protection likely requires a local vision model for element identification.
