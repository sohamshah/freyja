Runtime Orchestration: a Substrate for Agentic Organizations

01 / 11The problem with single-session agents

An LLM session is a closed box. The agent learns nothing between turns that wasn't in its system prompt; it sees nothing about other sessions that didn't get hand-copied into the new prompt; and it cannot tell you that this exact write happened in a peer session twenty seconds ago. Single-session agents are amnesiac and uncoordinated.

Multi-session agent systems built on top of this scaffold work around it three ways, all expensive:

  • Long context. Pile everything into a 200K window. Pays the prefill tax on every turn, dilutes attention, breaks past ~70% utilization on most models.
  • Shared memory blobs. A summarized state document that every session prepends. Works until the summary diverges from reality (no audit trail), or two sessions race to update it.
  • Manual coordination. The human operator copies a fact from session A into session B's prompt. The human becomes the substrate.

None of these compose. None of them give the team a single source of truth. None of them tell you, three weeks later, why the agent made a particular call — let alone whether two agents racing on the same artifact happened by design.

The reframe
Treat the project itself as the unit of memory, not the session. Every action — every tool call, every decision, every briefing, every question — lands as an event in an append-only log keyed on (tenant, project, session, actor). Every view a session needs is a projection over that log. Cross-session memory becomes a query, not a copy. Coordination becomes a hook, not an etiquette.

02 / 11The substrate: an event log + projections

The Phase 1B substrate is exactly two primitives. An append-only event log in Postgres, and a set of projectors that compute views over it on demand.

The event

Every event is the same shape:

Event · agent_harness/events/event.py
@dataclass(frozen=True)
class Event:
    event_id:    str             # evt_<hex>
    type:        str             # "decision.recorded", "tool.requested", ...
    actor:       Actor           # (kind, id, name) — flows from session.user_*
    session_id:  str
    project_id:  str | None      # scope for projections
    tenant_id:   str | None
    caused_by:   str | None      # event_id of the parent — forms the causal chain
    occurred_at: datetime
    payload:     dict[str, Any]  # type-specific
    prompt_hash: str | None      # content-hash registry pointer
    idempotency_key: str | None

Fourteen event types cover Phase 1. Each is emitted at exactly one place in the codebase, and each is consumed by at most one or two projectors. The discipline is what keeps the contract small.

The projector for the project's headline view — project_state — folds these into a single deterministic dataclass that the briefing pass and the operator UI both read from.

Slice · agent_harness/projections/project_state.py
@dataclass(frozen=True)
class ProjectStateSlice:
    project_id:                  str
    members_active_this_week:    list[str]
    recent_decisions:            list[Decision]       # last 14d, sorted desc
    in_flight_claim_event_ids:   list[str]            # active claims
    open_questions:              list[OpenQuestion]
    canonical_artifacts:         list[ArtifactRef]
    blockers:                    list[Blocker]
    suggested_next:              list[Suggestion]
    slice_hash:                  str                  # cache key
    # Phase 1B extensions — derived signal beyond decisions:
    recent_file_reads:           list[FileTouch]      # last 15, deduped
    recent_file_writes:          list[FileTouch]      # last 15
    recent_briefings:            list[BriefingRef]    # last 3 (prose preview)
    recent_interventions:        list[InterventionRef] # last 5
    recent_questions:            list[QuestionRef]    # last 10

The projector is pure: same events in, same slice out. Cached by slice_hash, which only changes when the input event set changes. No materialized table, no background job — the slice is computed on every read in <500ms for projects up to ~10k events.

Why projections, not tables
A materialized state table is a second source of truth that can drift. A projection over an append-only log cannot. If the slice ever looks wrong, you re-derive it from the events and the bug is in the projector, not in 800 places where state got updated.

03 / 11Profiles: orchestrator config that outlives any session

A profile is the versioned, project-scoped configuration that any session in the project binds against on entry. It's the persistence layer for orchestrator identity. One (tenant, name) pair has many versions; exactly one is published at a time.

FieldPurpose
base_instructionsThe system prompt the agent walks in with. Carries the orchestrator's voice + operating model.
default_model / reviewer_modelPrimary inference model and (future) reviewer for the reviewer pattern.
skill_foldersReferences to skill folders the runner loads into the prompt by scope (profile, tenant). Wiring lands in Phase 2.
named_inputsInputs the session must supply at create-time. Each has a required flag — missing required inputs return HTTP 400 with the missing keys.
named_outputsStructured returns the calling FA expects (text, generated_doc[], etc.).
tool_rosterauto_nominate flag + standard_tools list + allow/deny lists. Defines which tools the agent gets.

Profiles are themselves event-sourced: profile.created, profile.published, profile.archived events land in the same log. Forking, diffing, rolling back, and replay are all the same substrate primitive used everywhere else.

04 / 11Session entry: bind, snapshot, brief

When the FA creates a session against the substrate-enabled harness, three things happen in SessionManager._attach_runtime before the agent ever touches the user's first message.

POST /api/sessions
→
bind_profile
→
validate_named_inputs
→
snapshot_docs
→
emit session.opened
→
briefing turn (next)

Bind. The session's context carries profile_name. The hook resolves it via the profile store, pins runtime.profile_id / profile_version, and — when the FA didn't override the system prompt — fills session.system_prompt from profile.base_instructions.

Validate. For every NamedInput with required=True, the operator's session-create body must include a value under context.named_inputs. Missing required inputs raise ProfileInputError which the route maps to HTTP 400 with the missing keys listed. This is the FA's contract bug, not a runtime failure.

Snapshot. The docs folder is read once: plan + program_memory + the 10 most-recent generated_docs for this project. The snapshot lands on runtime.docs_snapshot so the prompt assembler can render it without re-querying.

Idempotency by accident
None of the three phases mutate state outside the session being created. Re-issuing the same session-create request lands a new session row but produces the same downstream events (with new ids). The event log is what carries continuity, not the session table.

05 / 11The briefing turn — deterministic slice, then prose

This is the substrate's headline feature. The first time a session sends a message, before the agent processes that message, the runtime runs a separate LLM call against a structured slice of the project's history. The output is a ≤200-word briefing plus a ranked suggested_next list. The briefing prose is appended to the session's system prompt before the user's turn runs. The agent walks into the user's first message already oriented.

Briefing pipeline — 5 stages, one substrate read, one substrate write

The slice is the deterministic half — same events, same slice, no LLM. The LLM is constrained to call record_briefing(prose, suggested_next[]) exactly once, with structured args parsed by parse_briefing_tool_call. The system prompt for the briefing pass requires the model to reference at least one signal from each non-empty slice category, so generated briefings reflect the actual project pulse — not generic "the project is going well" filler.

Sample briefing — what the agent walks in with
## Session briefing

Router project sits between cascade v2 landing and v3 design. Taxonomy axis enums
have been ratified (intent × complexity × domain) per dec_seed_001 and dec_seed_002;
the (transform × mixed) cell needs a finer label but that's deferred. Cascade v2 is
implemented in router/cascade_v2.py and passed offline replay against v1
(dec_seed_003). The next question on the table is the live grader model — opus,
sonnet, or ensemble — and the sampling rate for cascade-v3 rollout. Router Lead
has been the only active member this week. No published canonical artifacts yet —
everything is in working files. No blockers logged. The single open question is
evt_seed_q1 (grader sampling rate).

### Suggested next moves
- **iterate** → `dec_seed_003` — land cascade v2
- **answer_open_question** → `evt_seed_q1` — decide on grader model

The injection is what closes the loop. Before this work, the briefing was an audit event — visible in the log, invisible to the agent. Now format_briefing_for_prompt(BriefingOutput) → str renders the briefing as a markdown block and the runner appends it to both managed.harness_session.system_prompt (the live value the runner reads) and managed.system_prompt (the persistent attr, persisted via db.update_system_prompt so the briefing survives restarts).

06 / 11The tool-execute hook + duplicate-check

Every tool call goes through ToolRegistry.execute. The tool-execute hook wraps it with three substrate operations: emit tool.requested before, run the duplicate-check for artifact-creating tools, emit tool.responded after.

Tool-execute hook · duplicate-check flow

The duplicate-check pulls active claims from the in-flight registry projector and runs match_claims(intent, expected_outputs, snapshot, excluding_session_id). The match is exact on artifact name in expected_outputs — paths for write_file, artifact ids for generate_artifact, etc. A claim from another session whose expected outputs overlap with ours produces an intervention.surfaced event and a structured ToolResult with {"intervention": "duplicate_claim", "recommendation": ..., "matches": [...]} — the tool never runs.

This is the substrate's anti-coordination-bug primitive. Two sessions racing to draft router/cascade_v3.py don't both produce conflicting versions; the second session's agent sees the intervention payload, narrates the conflict, and (a good agent) defers or coordinates.

Why the hook lives in the tool registry, not in each tool
Wrapping the registry's execute() means every tool — built-in, MCP, custom — gets the same substrate semantics. The tools themselves don't know they're being audited. New tool? Free observability, free duplicate-check.

07 / 11read_project_state — the agent queries the substrate

Up to this point, the substrate flows information into the agent only via the briefing — a one-shot summary appended to the system prompt. The agent has to trust the prose, and can't verify a claim mid-turn ("did we really decide on opus as the grader?"). This is brittle.

read_project_state is a built-in tool that returns the live project_state slice JSON for the session's project — same shape the briefing turn consumes, materialized fresh on every call. The agent introspects the substrate during its turn instead of trusting a snapshot.

Tool schema
read_project_state(
  fields:               list[str]  # optional — restrict to a subset
                                   # (recent_decisions, recent_questions, etc.)
  max_items_per_list:   int        # cap each list, most-recent kept (default 20)
)
→ JSON {
    project_id, slice_hash,
    members_active_this_week,    recent_decisions,
    in_flight_claim_event_ids,   recent_file_reads,
    recent_file_writes,          recent_briefings,
    recent_interventions,        recent_questions,
    canonical_artifacts,         open_questions,
    blockers,                    suggested_next
  }

The profile's base_instructions tells the agent to call read_project_state before any non-trivial routing/grading/taxonomy decision — verify what the briefing said, cite the actual decision_id in the response, look up prior conflicting decisions before recording a new one. The substrate becomes alive in the agent's reasoning, not a passive context appendage.

Closing the loop
Briefing flows the project's history into the agent at session-start. read_project_state lets the agent flow questions back into the substrate during the turn. Together they make cross-session memory bidirectional.

08 / 11Multi-actor identity flows through the log

Every event carries an Actor(kind, id, name). The actor is derived from the session's user identity at session-create time, and propagates through every event the session emits. The project-state slice then rolls per-actor signals up:

  • members_active_this_week — distinct actor.id over the last 7 days
  • recent_decisions[].decided_by_actor_id — who recorded each decision
  • recent_file_reads[].by_actor_id / recent_file_writes[].by_actor_id
  • recent_questions[].asked_by_actor_id — who raised each HITL prompt
  • recent_interventions[].matches[].actor — who's holding the conflicting claim

Identity flows from the session-create request body when auth is disabled (body.user_id, body.user_name, body.user_email) and from JWT claims when it isn't. The substrate doesn't care which path — it just attaches whatever the session presents to every emitted event.

The runtime console exposes this via the user switcher pill in the topbar. Click → pick from a curated roster of 8 identities (5 ema, 3 kpmg). Subsequent sessions you create are attributed to that user. Each actor gets a deterministic color (HSL hash on actor.id) so collaboration becomes visible in the slice rendering.

09 / 11The runtime console — every primitive made visible

The substrate is invisible by design. Hooks, projectors, event logs — none of these surface on their own. The runtime console at /runtime_console is the operator's instrument: a single-page surface that makes profile config, session creation, multi-actor chat, and every projection visible, and makes the qualitative tests of the substrate's value props clickable.

What this earns
Every claim in the Phase 1 spec ("projects compound", "sessions coordinate", "decisions outlive their session", "every action is auditable") moves from being a closing line to being a clickable test. The runtime console is the instrument; the substrate is the moat.

11 / 11What remains

Phase 1B closed every move on the original plan but left calibrated gaps. Some are out-of-scope Phase 2 features; others are real Phase 1 work that didn't make this branch.

Still in Phase 1 scope

  • Skill loading. Profiles declare skill_folders but the runner doesn't load them. Scaffolding exists; loader is a half-day of work.
  • Reviewer pattern. Profiles have a reviewer_model field but no review hook fires. The spec wants a per-turn or per-doc reviewer pass.
  • Artifact rendering. When the agent calls generate_artifact, the chat shows a generic tool card. Could be a dedicated artifact card that links to the docs folder entry.
  • BYOM exercise. Secret Manager integration is wired but no end-to-end test against a tenant secret.
  • blockers projector. Slice has the field but no emitter — would scan decision titles prefixed blocker:.

Phase 2 features (deliberately out)

  • Knowledge-gap signal collection (low-confidence outputs feed the taxonomy)
  • Cost budget enforcement with auto-reroute
  • Live grading + reviewer pool
  • Deploy-gate replay regression
  • Sub-agent delegated identity (chained actor)
  • Decision lineage (supersedes, blocked_by)

UI ergonomics not yet built

  • Profile edit (workaround: re-seed with force=true which archives the prior version)
  • Profile diff between versions
  • Session detail panel — pick any past session and see its full timeline + system prompt + messages in one view
  • "Open peer session" button (manual today)
  • Tenant analytics — cross-project view of decisions made, sessions run, intervention rate
  • Export turn/session as transcript

The deeper unfinished business

The two things that, when shipped, will make the substrate feel finished rather than merely working:

  1. Skill evolution. Right now profiles are hand-edited and seeded. Phase 2's observation pipeline watches low-confidence outputs, proposes skill folders that close knowledge gaps, and a curator-style review queue ratifies the additions. The substrate already records the signal (briefing.failed, low-confidence tool results), so the loop is closeable.
  2. Replay regression. The event log is replayable by construction. Phase 2's deploy gate replays last quarter's prod sessions against a candidate profile/orchestrator and surfaces the decision diff. Held-or-pass. This is the Sarbanes-Oxley / EU AI Act primitive — not a static eval suite, but "we replayed prod behavior against the candidate and the diff is bounded."

Both compose with what's already shipped. Both depend on the event log being authoritative, which it is. The runtime console gives the operator the surface to drive them when they arrive.
